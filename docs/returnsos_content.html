Problem Framing

The Reverse Logistics Challenge

The Ask
When a returned device lands in a warehouse, an operator must instantly decide: do we Resell it, Repair it, Cannibalize for parts, or Recycle it?
The Difficulty
Every device is a snowflake. A cracked screen on an iPhone 14 Pro means "Repair". The exact same cracked screen on an iPhone 11 means "Cannibalize".
The Result
Operators rely on tribal knowledge. Processing is slow, outcomes are wildly inconsistent, margins are lost, and safety constraints (like swollen batteries) are occasionally bypassed.
Speaker Script (0:30 - 2:30)
"What problem did I choose? I focused on the decision bottleneck at the receiving dock. When a device arrives, there are multiple possible paths: resell, repair, harvest for parts, or recycle.

Why is this hard? Because the economics change drastically based on the specific model, its current grade, and real-time repair costs. An operator can't do the math in their head. A cracked screen on a premium model justifies a $120 repair. On an older model, that same repair guarantees a financial loss.

Today, warehouses rely on tribal knowledge. This leads to slow processing, inconsistent outcomes, and ultimately, millions of dollars in lost recovery margin. ReturnsOS replaces that guesswork with a deterministic and AI-assisted decision engine."

"Moving into System Design. ReturnsOS is built using a decoupled architecture to ensure scalability.

On the frontend, we use React to create the Operator Console and the Supervisor Decision Hub. We recently upgraded this with a premium light-mode default, dual-column configurators, and rich toast notifications.

The backend is built on FastAPI. I split the logic into distinct functional areas: a Device Service for intake, a Rules Engine for constraints, a Valuation Service for economics, and an AI Service. For data, I used SQLite to make local development seamless, but the schema is designed to map directly to PostgreSQL in production."

Speaker Script (4:00 - 6:00)
"Let's trace the data flow.

First, an operator inputs the device diagnostics. The Device Service ingests this and emits an event.

Next, the Rules Engine intercepts the payload. It applies a 'Lowest-Grade Rule' and checks for absolute hard constraints—like a swollen battery.

Then, the Valuation Service kicks in. It pulls dynamic base market values and component repair costs from the database to map out the gross and net yield of every single disposition channel.

Finally, this data is passed to the AI Decision layer which identifies the optimal path, resulting in a finalized recommendation presented to the warehouse supervisor."

Tradeoffs

Key Decisions & Alternatives

Tradeoff 1 Rules Before AI
Decision: Run deterministic business rules before invoking the AI agent.
Alternative: Let an LLM handle all routing.
Why: Safety decisions (like swollen batteries) must never be probabilistic. AI is expensive and can hallucinate; hard rules guarantee absolute safety compliance.
Tradeoff 2 SQLite vs PostgreSQL
Decision: Shipped with SQLite locally.
Alternative: Force a Dockerized Postgres database.
Why: Reduced friction for evaluators. The schema (JSONB columns) is strictly designed to migrate to Postgres simply by swapping the SQLAlchemy connection string.
Tradeoff 3 SSE over WebSockets
Decision: Used Server-Sent Events for the event stream UI.
Alternative: Bi-directional WebSockets.
Why: The live event feed is read-only from server to client. SSE is lighter, runs over standard HTTP, and has automatic reconnections built-in.
Speaker Script (8:00 - 10:00)
"Let's discuss Tradeoffs. Clear thinking matters more than perfect diagrams, so here are three major architectural decisions I made.

First: Rules before AI. Why didn't I just build an LLM wrapper? Because safety compliance cannot be probabilistic. If a battery is swollen, a deterministic rule immediately routes it to recycling. The AI is only consulted for complex economic optimizations.

Second: Shipping with SQLite instead of PostgreSQL. I wanted zero friction for anyone reviewing the code. However, I deliberately used JSONB data structures so that transitioning to Postgres in production requires zero schema changes.

Third: Using Server-Sent Events instead of WebSockets. The event log is a one-way broadcast from the backend. SSE is much lighter and natively handles reconnections without the overhead of maintaining a WebSocket state."
Tradeoffs

What Would Change My Mind?

Scale & Multi-Tenancy
If we onboard 50 different warehouse clients, maintaining hard rules per client becomes a bottleneck. I would shift to a multi-tenant DB with Row-Level Security, and lean heavier into Fine-Tuned ML classification models over manual rule configs.
Message Bus Evolution
Currently, the event bus is simulated in memory. If throughput hits thousands of devices per hour, I would immediately swap the memory bus for an actual Kafka or RabbitMQ cluster to ensure message durability and backpressure handling.
Speaker Script (10:00 - 11:30)
"What would change my mind on these architecture choices?

Scale. Right now, this is a single-tenant MVP. If we had to support 50 different warehouse clients, maintaining distinct hard rules for each one would become a massive bottleneck. I would immediately implement Row-Level Security in Postgres, and I would shift away from manual rules towards a fine-tuned Machine Learning classification model trained on historical audit data.

Also, as intake volume scales to thousands of devices per hour, my simulated event bus would drop messages. I would change my mind on infrastructure simplicity and integrate an actual Kafka cluster to guarantee message durability and handle backpressure."

Speaker Script (11:30 - 12:30)
"Let's look at Failure Modes. Since we haven't integrated a live production LLM yet, what logic actually breaks today?

The biggest vulnerability is data dependency. For example, if a supervisor accidentally deletes the base market value for an 'iPhone 14' in the Rule Configurator, the Valuation Service will fail to find a match. It will fall back to an arbitrary default, destroying our Recovery Value accuracy.

Another edge case: Conflicting priorities. What if a device is both Activation Locked and has a Swollen Battery? If our rule execution order is flawed, the system might try to route it to Lock Clearance—wasting money on a device that is legally required to be recycled. Rule hierarchy is critical."
Failure Modes

System Fallbacks

Graceful Degradation
If the AI service times out or hallucinate an invalid schema, the system catches the error and degrades gracefully to the deterministic Rules Engine. The device is flagged for manual supervisor review.
Safety Absolute
Safety rules always bypass AI. A swollen battery is routed to Recycle before the AI service is even queried.
Speaker Script (12:30 - 13:30)
"How does the system behave when things go wrong? It degrades gracefully.

If we do plug in a live LLM and the API times out, or it returns a malformed JSON schema, ReturnsOS catches the error. It automatically falls back to the deterministic Rules Engine and flags the record for manual supervisor review.

We never allow a system failure to compromise safety. The safety constraints fire before any external service is called, ensuring that a hazardous battery never ends up in a repair queue just because an API went down."
