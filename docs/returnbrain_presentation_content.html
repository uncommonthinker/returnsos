03 · Architecture

The decision flow

1
Operator intakes device
2
DeviceIntaked event emitted
3
Rules Engine evaluates
4
AI Decision Service invoked
5
Recommendation ready
6
Operator confirms
Decision logic inside steps 3 + 4
Hard rule?
Rule fires → action set deterministically; AI is skipped entirely. Used for safety-critical and compliance decisions.
Soft rule?
Rule context injected into AI prompt as guidance. AI reasons within the policy envelope but makes the final call.
No rule
AI has full autonomy. Confidence score governs whether output goes to auto-queue or human review.
Speaker note
Anticipated question: "Why rules before AI?" Answer: Because safety and compliance decisions cannot be probabilistic. Rules are deterministic by design. Hard rules are never bypassed by AI — this is structural, not just a prompt instruction.
03 · Architecture — AI layer

Four-agent AI pipeline

1
Condition Assessment
Normalises raw device signals into a structured condition summary. Removes ambiguity before the decision is made.
→ condition_summary
2
Recovery Valuation
Estimates recovery value across all five disposition paths. Does the economics before the recommendation.
→ valuation_map
3
Disposition Recommendation
Selects optimal path from condition summary + valuation map + rule context. The primary decision agent.
→ recommendation
4
Audit Agent
Verifies recommendation is self-consistent and rule-compliant. Catches hallucination before output is stored.
→ approved / flag
Prompt engineering guardrails
Temperature
0.0 — deterministic output for production decisions
Schema enforcement
Pydantic validation; retry once with correction prompt on fail
No PII to LLM
Only intake_id sent to Claude — no serial numbers or owner data
Audit hash
SHA-256 of full prompt + response stored on decision record
Speaker note
Four agents, not one. Separation of concerns: each agent is testable independently. If valuation logic changes, you don't touch the audit agent. The audit agent is what catches the "confident but wrong" scenario before it reaches the operator.
03 · Architecture

Why it's designed this way

ADR-001 Why FastAPI?
Async-native for non-blocking LLM calls. Pydantic v2 enforces request/response schemas at runtime. Auto-generated OpenAPI docs. Strong Python type hints make service contracts explicit.
ADR-002 Why RabbitMQ over Kafka?
v1 volume is modest — RabbitMQ is operationally simpler with lower resource overhead. Events designed to be Kafka-compatible (keyed by device_id); migration in v2 is a protocol swap, not a schema redesign.
ADR-003 Why Anthropic Claude?
Structured output enforcement. Strong chain-of-thought reasoning. Provider isolated behind a DecisionLLMProvider interface — swap in GPT-4o with one adapter class change.
ADR-004 Why rules before AI?
Hard rules are deterministic. AI is probabilistic. Safety-critical decisions must never be probabilistic. Pre-evaluation also reduces wasted LLM cost on deterministic cases.
ADR-005 Why PostgreSQL only — no polyglot?
Single engine for ACID + JSONB (rule conditions, AI reasoning). Eliminates operational complexity. Rule conditions are read-parsed into Python objects at startup; complex querying happens in application layer.
Speaker note
For each ADR be ready to go deeper. Track B live discussion will probe: "Why RabbitMQ and not Kafka?" and "Why not a single monolith?" Have crisp answers ready. The "Rules before AI" point almost always gets a follow-up — nail it.

04 · Limitations

What's missing — deliberately

L-01
Market price is manual
Operator types in reference price. Stale data leads to suboptimal recommendations.
→ v2: Scheduled scraper from Cashify, OLX, eBay India
L-02
No learning loop
AI does not improve from past decisions. Model is static once deployed.
→ v3: Fine-tune on outcome data. Reinforcement learning from recovery results.
L-03
No WMS integration
Device records entered manually or via CSV. Double entry risk.
→ v2: REST webhook integration with common WMS providers
L-04
No visual grading
Cosmetic grade (A/B/C/D/F) relies entirely on operator judgment.
→ v2: Computer vision model to auto-grade from intake photos
L-05
Single tenant
Database and services are single-tenant. Cannot serve multiple warehouse clients.
→ v2: Tenant-aware data model with row-level security
L-06
No auto-execution
All decisions require operator confirmation. Human in every loop.
→ v2 (when trust is earned): Auto-execute above confidence 95%
Speaker note
Damco wants self-awareness. Do not apologise for limitations. Frame them as deliberate scoping decisions. "I know exactly what's missing and why I left it for v2." Every limitation here has a clear fix path. This is not a weakness — it is a roadmap.
